# V2 Milestone 2.1 Plan — SQLite Label Name Mapping (Minimal)

## Goal
Write readable `X-Keywords` label names in archived downloads, using SQLite as the only label dictionary.

## Constraints
- Keep implementation minimal.
- No in-memory label map/cache.
- Resolve labels via SQL lookups.

---

## Task 1: Schema

Files:
- `lib/gmail/list_pages_schema.go`
- `lib/gmail/*_test.go`

- [ ] Add table:
  - `gmail_labels(id PRIMARY KEY, name, type, updatedAtMs)`
- [ ] Ensure schema creation/migration is safe.
- [ ] Add schema test coverage for `gmail_labels`.

## Task 2: SQL-only label resolution path

Files:
- `lib/gmail/message_sync.go`
- `lib/gmail/gmail.go`
- `lib/gmail/*_test.go`

- [ ] For each message label ID, resolve via SQL:
  - `SELECT name FROM gmail_labels WHERE id=?`
- [ ] If label ID not found, temporarily fallback to raw ID.

## Task 3: Lazy refresh on unknown label ID

Files:
- `lib/gmail/message_sync.go`
- `lib/gmail/service.go`

- [ ] If any unknown label ID is encountered while mapping one message:
  - fetch `Users.Labels.List`,
  - upsert all labels into `gmail_labels`,
  - retry SQL mapping for that message.
- [ ] Guard refresh so it happens at most once per run unless explicitly needed.

## Task 4: Header output behavior

Files:
- `lib/gmail/message_sync.go`

- [ ] Write `X-Keywords` using resolved names when available.
- [ ] Keep fallback raw label ID when still unresolved after refresh.
- [ ] Preserve existing multi-header behavior.

## Task 5: Verification

- [ ] Unit test: known ID maps to readable name.
- [ ] Unit test: unknown ID triggers lazy refresh and then maps.
- [ ] Unit test: still-unknown ID stays raw.
- [ ] Full suite: `go test ./...`

## Notes
- This milestone intentionally favors simplicity over micro-optimization.
- SQL lookups are acceptable for now; optimize later only if measured bottleneck.
